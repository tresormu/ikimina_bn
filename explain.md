# IkiminaPass Backend Documentation

## 1. File and Directory Architecture & Purpose

This section outlines every file in the `src` directory and explains its primary responsibility within the IkiminaPass backend system.

### Core Configuration & Bootstrapping
*   **`main.ts`**: The entry point of the NestJS application. It bootstraps the server, configures global validation pipes (to ensure all incoming requests meet DTO constraints), registers the global exception filter for unified error handling, configures CORS, and sets up Swagger for API documentation.
*   **`app.module.ts`**: The root module that ties all feature modules together. It also configures global utilities like `@nestjs/config` for environment variables, `@nestjs/schedule` for Cron Jobs, and the `ThrottlerModule` (`@nestjs/throttler`), which enforces a strict rate limit of 100 requests per minute globally for enhanced security.
*   **`app.controller.ts` / `app.service.ts`**: Default boilerplate files generated by NestJS (usually kept for simple health checks or ping endpoints).
*   **`package.json`**: Defines the project's runtime dependencies and available npm scripts (e.g., `start:dev`, `start:prod`, `test:e2e`).
*   **`.env.template`**: Documents all required environment variables needed to run the project, including `DATABASE_URL`, `JWT_ACCESS_SECRET`, `JWT_REFRESH_SECRET`, Africa's Talking credentials (`AT_USERNAME`, `AT_API_KEY`), and MTN MoMo credentials (`MOMO_SUBSCRIPTION_KEY`, etc.).
*   **`docker-compose.yml`**: Spins up the PostgreSQL 15 database container required for local development and testing.

### Database Management (`src/prisma/`)
*   **`prisma/schema.prisma`**: The single source of truth for the entire database. It defines all models, enums, and relations used throughout the platform (e.g., User, Group, Subscription, AuditLog).
*   **`prisma.service.ts`**: Connects the application to the PostgreSQL database using Prisma Client. It manages the connection lifecycle (`onModuleInit` and `onModuleDestroy`).
*   **`prisma.module.ts`**: A global module that exports the `PrismaService`, allowing any other module in the application to access the database without needing to import it redundantly.

### Common Utilities & Security (`src/common/`)
*   **`decorators/current-user.decorator.ts`**: A custom parameter decorator that extracts the authenticated user from the request object, making it easy to access `user.id` in controllers.
*   **`decorators/public.decorator.ts`**: Marks specific endpoints as "public" so the global JWT guard bypasses authentication for them (e.g., login, register, USSD webhooks).
*   **`decorators/roles.decorator.ts`**: Allows attaching required roles (e.g., `ADMIN`, `TREASURER`) to specific routes.
*   **`filters/global-exception.filter.ts`**: Catches all unhandled exceptions and formats them into a standardized JSON response (`{ statusCode, errorCode, message, timestamp, path }`), ensuring the frontend receives consistent error structures.
*   **`guards/jwt-auth.guard.ts`**: Enforces JWT token validation on routes. If a route isn't marked `@Public()`, this guard rejects unauthorized requests.
*   **`guards/roles.guard.ts`**: Checks if the authenticated user has the necessary roles specified by the `@Roles()` decorator.
*   **`AuditLog Model`**: Used extensively across the platform to track sensitive actions such as admin interventions, USSD complaints, lender report flagging, and subscription overrides, providing a comprehensive and immutable historical trail.

### Feature Modules

**Auth Module (`src/auth/`)**
*   **`auth.controller.ts`**: Exposes endpoints for sending OTPs, verifying OTPs, refreshing tokens, and logging out.
*   **`auth.service.ts`**: Contains the business logic for hashing/verifying OTPs (using bcrypt), enforcing SMS rate limits, and securely issuing JWT Access and Refresh tokens.
*   **`jwt.strategy.ts`**: Configures Passport to extract the JWT from the Authorization header and validates the user against the database.
*   **`dto/*.dto.ts`**: Defines strict validation rules for phone numbers, OTP lengths, and token structures.

**Users Module (`src/subscriptions/users/`)**
*   **`users.controller.ts`**: Handles profile retrieval, profile updates, and account soft deletions.
*   **`users.service.ts`**: Interacts with the database to fetch user details, update fields like `fullName` and `nationalId`, and manages the soft-delete (`isActive: false`) logic to preserve data integrity.
*   **`dto/update-user.dto.ts`**: Validation for optional profile updates.

**Groups Module (`src/groups/`)**
*   **`groups.controller.ts`**: Endpoints for group creation, joining via invite codes, updating rotation schedules, and posting announcements.
*   **`groups.service.ts`**: Manages the core ROSCA logic. Handles the generation of invite codes, manages the `GroupMember` relationships, creates initial free-trial subscriptions upon group creation, and tracks rotation changes.
*   **`dto/*.dto.ts`**: Validates inputs for creating groups, updating settings, and modifying member rotations.

**Subscriptions Module (`src/subscriptions/`)**
*   **`subscriptions.controller.ts`**: Exposes endpoints for checking group tiers, viewing billing history, and initiating manual payments.
*   **`subscriptions.service.ts`**: The core monetization engine. It calculates the group's tier based on member count (`STARTER`, `GROWTH`, `COMMUNITY`, `ENTERPRISE`), handles manual MoMo payment workflows, and runs automated daily Cron jobs for billing, grace periods, and group suspensions.

**Contributions Module (`src/contributions/`)**
*   **`contributions.controller.ts`**: Allows members to submit MoMo transaction IDs for their weekly payments and view their history.
*   **`contributions.service.ts`**: Validates MoMo transactions, tracks weekly deadlines, and calculates who is owed the pot for the current week based on the rotation log.

**Credit Score Module (`src/credit-score/`)**
*   **`credit-score.controller.ts`**: Endpoints for users to view their score, share reports, and for public access to shared reports.
*   **`credit-score.service.ts`**: A complex calculation engine that evaluates tenure, on-time payment percentages, missed payments, and loan repayment history to generate a score between 0 and 850. Handles the creation and revocation of temporary, shareable UUID links.

**Loans Module (`src/loans/`)**
*   **`loans.controller.ts`**: Handles loan requests, democratic voting, and treasurer repayments.
*   **`loans.service.ts`**: Enforces the 6-month tenure requirement for requesting loans. Tallies `APPROVE`/`DECLINE` votes from members, automatically disburses funds if majority approval is reached, and tracks repayments.

**Disputes Module (`src/disputes/`)**
*   **`disputes.controller.ts` & `disputes.service.ts`**: Allows members to raise issues regarding payments or group operations. Treasurers can view and resolve these disputes, or escalate them to platform Administrators if consensus cannot be reached.

**Lenders Module (`src/lenders/`)**
*   **`lenders.controller.ts` & `lenders.service.ts`**: Allows financial institutions to register and authenticate. Enables approved lenders to search for members by phone or National ID and securely view their full credit reports—provided the member belongs to a group with an active subscription.

**Notifications Module (`src/notifications/`)**
*   **`notifications.controller.ts` & `notifications.service.ts`**: A centralized messaging hub. Stores notification records in the database, manages user notification preferences (opt-in/opt-out), and attempts to send SMS/Push notifications (mocked in development).

**USSD Module (`src/ussd/`)**
*   **`ussd.controller.ts` & `ussd.service.ts`**: Provides a webhook for Africa's Talking. Parses plain text inputs from feature phones, tracks session states, and responds with dynamic text menus allowing users to check their contributions and credit scores entirely offline.

**Admin Module (`src/admin/`)**
*   **`admin.controller.ts` & `admin.service.ts`**: Dedicated high-privilege endpoints for platform operators to monitor revenue, override suspended subscriptions, approve/reject lender applications, and broadcast system-wide announcements.

## 2. How the IkiminaPass Backend Works

IkiminaPass is a robust, production-ready backend built with NestJS, TypeScript, and Prisma (PostgreSQL). It acts as the central brain for a digitized Rotating Savings and Credit Association (ROSCA) platform. The system operates on a highly secure, stateless architecture using JWT (JSON Web Tokens) for authentication. Every request requires an Access Token, while long-lived sessions are maintained securely via Refresh Tokens. 

The database leverages strict UUID primary keys and relational integrity to ensure that financial data, such as contributions, loans, and credit histories, are inherently linked to users and their respective groups. Data is never permanently erased; instead, the system utilizes "soft deletes" (marking records as inactive) to maintain a complete, immutable audit trail required for financial compliance and accurate credit scoring.

The architecture is fundamentally modular. Separation of concerns is strictly enforced: Controllers handle HTTP routing and validation parsing, while Services contain the heavy business logic. Cross-cutting concerns like role-based access control (checking if a user is a `TREASURER` vs. a `MEMBER`) and unified error formatting are handled elegantly by NestJS Guards and Exception Filters. Furthermore, the system is fully automated; a suite of Cron Jobs (`@nestjs/schedule`) continuously monitors the database in the background to enforce weekly contribution deadlines, trigger billing cycles, send warning notifications, and automatically suspend groups that fail to pay their subscriptions.

## 3. Who Uses the System and What Can They Do?

The platform caters to four distinct user roles, each with specialized capabilities and restrictions enforced by the backend:

### **1. Members (The Core Users)**
Members use the mobile app or USSD interface to participate in savings groups.
*   **Authentication**: They log in exclusively via OTPs (One Time Passwords) sent to their phone numbers via Africa's Talking. As a deliberate security design, after a successful verification, all OTP records associated with that phone number are completely deleted from the database.
*   **Capabilities**: Members can join groups using 8-character invite codes. Once inside, they submit MTN MoMo transaction IDs to fulfill their weekly savings quotas. They can view the group's activity feed, track whose turn it is to receive the payout, and raise "Disputes" if they believe a mistake was made.
*   **Financial Leverage**: After 6 months of active, reliable contributions, members can request emergency loans. The backend also tracks their reliability to build an algorithmic Credit Score (0-850). Members can generate temporary, secure URLs to share their credit reports with external parties.

### **2. Treasurers (The Group Managers)**
Treasurers are members with elevated administrative rights over a specific group.
*   **Group Management**: They create the group, define the contribution amount (e.g., 5,000 RWF), set the frequency (Weekly/Monthly), and define the initial rotation order for payouts. They have the power to alter this rotation order later, with the backend quietly maintaining an audit log of every change.
*   **Oversight**: Treasurers mark members as inactive, post official announcements to the group feed, and resolve internal disputes. They also record manual repayments for approved loans.
*   **Billing Responsibility**: The Treasurer represents the group as the paying customer. They receive automated SMS warnings regarding the group's subscription status and are responsible for initiating MTN MoMo payments to clear overdue subscription balances and prevent the group from being suspended.

### **3. Lenders (Financial Institutions)**
Lenders are external entities (like banks or microfinance institutions) seeking reliable borrowers.
*   **Access**: They must register with their institution name and BNR license number, and wait for platform Admins to approve their account. They log in using an Email and Password.
*   **Capabilities**: Approved Lenders have a read-only perspective. They can search the platform using a phone number or National ID. If the queried user belongs to an active, paying group, the Lender can instantly view that user's full, algorithmically generated credit report, enabling them to make rapid, data-driven loan decisions outside the platform.

### **4. Administrators (The Platform Operators)**
Admins oversee the entire IkiminaPass ecosystem from a high-level dashboard.
*   **Platform Control**: They approve or reject new Lender registrations. They possess the ultimate authority to manually override group subscriptions (e.g., extending a free trial in exceptional cases) or force-suspend malicious groups.
*   **Dispute Escalation**: If a Treasurer cannot resolve a dispute within a group, it is escalated to the Admins for final review and resolution.
*   **Analytics**: Admins can query platform-wide statistics, viewing real-time subscription revenue breakdowns, tier distributions (Starter vs. Enterprise), and user growth metrics. They can also broadcast critical system announcements to all users simultaneously via push notifications.
